# 日報：全員用（D）＋ 代表日報（時間帯別） — プラン

**基本方針は A**（全員が書くもの ＋ 代表者が時間帯別に書くもの）。**全員用は D を採用**（チェック＋短いコメント、書く負担を減らす）。全員用の選択項目は**より個人に特化**した内容にする。代表日報と全員用は相談して書くわけではないため、**重複や漏れはある程度仕方ない**として許容する。

---

## 1. 前提の整理

| 項目 | 方針 |
|------|------|
| 構成 | **全員用** ＋ **代表日報（時間帯別）** ＋ **店長・マネージャー日報**（まとめ） |
| 全員用の形式 | **D**: チェックリスト ＋ 短いコメント。書く負担を減らす |
| 全員用の選択項目 | **個人に特化**（自分の体調・体感・困ったこと・良かったことなど） |
| 代表日報 | **時間帯別**: ランチ / アフタヌーン / ディナー などで分けて代表者が記入 |
| 店長・マネージャー日報 | **残す**。時間帯日報・全員日報を参照して、その日の**まとめ**を書く |
| 重複・漏れ | 相談して書くわけではないので発生しうる。ある程度許容する |

---

## 2. 全員用日報（個人特化・チェック＋短コメント）

### 2.1 目的

- その日出勤した**全員の声**を残す（パート・正社員問わず）
- **負担を小さく**する（チェック中心＋1行程度のコメント）
- **個人の状況・体感**に特化し、代表日報（店の状況）と役割を分ける

### 2.2 項目案（個人に特化した選択＋短コメント）

**チェック系（複数可 or 単一）**

| カテゴリ | 例（選択項目） |
|----------|----------------|
| 自分の体調・コンディション | 問題なし / やや疲れ / 体調不良気味 / その他 |
| 忙しさの体感（自分の担当範囲） | 余裕 / ちょうど良い / きつかった / かなりきつかった |
| 困ったこと・トラブル（自分が遭遇した） | 特になし / お客様対応 / 機械・品切れ / 人手不足 / その他 |
| 良かったこと・気づき（個人） | 特になし / 接客でうまくいった / 改善のアイデアあり / その他 |

**自由記述**

- **ひとことメモ**（1〜2行程度。任意 or 短い必須）

**※ 全員用の選択項目は上記で確定。**

### 2.3 データ・画面イメージ

- 1日1人1件（その日勤務した人が各自1回だけ入力）
- 保存: 日付・店舗・ユーザーID・チェック結果・短いコメント・入力日時 など
- 一覧: 日付・店舗でフィルタ。全員用は「その日の全員分」を一覧表示できるとよい

---

## 3. 代表日報（時間帯別）

### 3.1 目的

- **店の状況**を時間帯ごとに記録する（誰か1人が一日中いることが少ない前提）
- ランチ・アフタヌーン・ディナーなど**時間帯ごとに代表者**が1本ずつ書く

### 3.2 時間帯の定義・マスタ

- **時間帯マスタ**は別途定義・議論する（名称・開始・終了時刻など）。ここでは「ランチ／アフタヌーン／ディナー」を想定した項目定義のみ行う。

### 3.3 代表者の決め方・記入欄（要実装）

- **代表者は店長・マネージャーが事前に決める。**
- **記入欄（画面）を作る**: 日付・店舗を選び、各時間帯（ランチ／アフタヌーン／ディナー）ごとに「誰を代表にするか」を選択・保存する。
- **該当日のシフトを参照できるとよい**: 代表者を決める際、その日のシフト（出勤者・時間帯）を見ながら選べるようにする。シフト一覧 or カレンダー上で該当日のシフトを表示し、その上で各時間帯の代表を割り当てる。

（例: 2/15 の代表割り当て画面で、2/15 のシフト表を表示 → ランチ代表・アフタヌーン代表・ディナー代表をそれぞれ選択して保存。）

### 3.4 代表日報の項目定義

代表日報は「**その時間帯の店の状況**」を代表者が記入する。1日分の店長日報を時間帯別に分割するイメージ。以下を定義する。

| 項目 | 説明 | 形式 |
|------|------|------|
| 日付 | 対象日 | 日付 |
| 店舗 | 対象店舗 | 店舗ID |
| 時間帯 | ランチ／アフタヌーン／ディナー（時間帯マスタで定義） | 時間帯ID or コード |
| 代表者（記入者） | 誰が書いたか（事前に店長が決めた代表者 or 入力時選択） | ユーザーID |
| 売上・状況の評価 | その時間帯の売上・店の調子 | 選択（例: 目標超過／達成／ほぼ／未達、または とても良い／良い／普通／あまり良くない） |
| 店の状況 | 混雑・雰囲気・清掃など | 選択 or 短文 |
| 人手体制 | その時間帯の人手 | 選択（例: 余裕あり／ちょうど良い／ぎりぎり／不足） |
| 出来事 | クレーム、大口予約、トラブル、体調不良など（複数可） | チェック or 複数選択 |
| 特記事項・メモ | 自由記述 | テキスト（数行） |

※ 既存の「店長・マネージャー日報」の項目（売上評価・店舗状況・人員体制・出来事・テーマ・メモ）を時間帯単位にした形。必要に応じて「テーマ」を追加・削除可能。

### 3.5 データ・画面イメージ

- 1日あたり **時間帯ごとに1件**（ランチ1本・アフタヌーン1本・ディナー1本）
- 保存: 日付・店舗・時間帯・記入者・内容・入力日時 など
- 一覧: 日付・店舗でフィルタ。その日の「ランチ／アフタヌーン／ディナー」が並んで見られるとよい

---

## 4. 店長・マネージャー日報（まとめ）— 残す

### 4.1 位置づけ

- **店長・マネージャー日報は残す。** 時間帯日報と全員日報を参照して、その日のまとめを書く。**全員日報と時間帯別日報の要約を日報に張り付ける**（保存時に要約を埋め込む）。**本部はこの日報を見れば全て把握できる。**

### 4.2 要約の張り付け（基本B：統合1本）

- 店長・マネージャー日報に張り付けるのは、**全員日報と時間帯別日報を合わせた1本の要約**（**基本B**）。AIで両方の元データを統合し、事前定義項目で整理した要約を1ブロックとして日報に保存する。
- 本部が店長・マネージャー日報を開けば、まとめ文＋統合要約1本で一覧でき、**他画面を見にいかずに全て把握できる**。

### 4.3 日別売上の参照（Look up）

- 店長日報入力時に、**その日の日別売上**を参照（look up）できるようにする。
- **入力時点で日別売上が未入力なら 0 でもよい。** 後から売上を入力した場合、**本部が閲覧する時点ではその日の日別売上を参照して表示**すれば、最新の売上が反映される。
- 実装イメージ: **閲覧時にその日の日別売上APIを参照して表示**する方式なら、本部が見る時は常に最新の売上になる。

### 4.4 入力時の参照は元データのみ

- 店長日報を**書く**画面では、その日の時間帯日報・全員日報の**参照は元データのみ**でよい。元データを見ながらまとめを記入し、保存時にシステム側で**全員＋時間帯を統合した1本の要約**を生成して日報に張り付ける。

---

## 5. ダッシュボード・ヘッダー（本部）

- **ダッシュボードには全員日報と時間帯別日報は置かない。店長・マネージャー日報のみ**をダッシュボードに載せる。本部は店長・マネージャー日報（要約張り付け済み）を見れば全て把握できる。
- **全員日報・時間帯別日報を直接見たければ、ヘッダーから遷移して確認**する（一覧・詳細画面へはヘッダーのリンクでアクセス）。
- **本部のヘッダーはすべての権限を網羅**する。必要な機能（ダッシュボード・日報・店長日報・全員日報・時間帯日報・シフト・売上・お知らせ・テナント募集 など）へすべてヘッダーから遷移できるようにする。現状で足りていればそのまま、不足があれば追加する。

---

## 6. 重複・漏れについて（全員用・代表日報どうし）

- 全員用と代表日報は**相談して書く前提ではない**
- **重複**（同じ出来事を両方で書く）や**漏れ**（どちらにも書かれない）は起こりうる → **ある程度仕方ない**として許容
- 本部は**店長・マネージャー日報（要約張り付け済み）を見れば全体を把握**。必要ならヘッダーから全員日報・時間帯別日報の元データへ飛べる。

---

## 7. 日報要約（3種類）— 店長日報への張り付けは統合1本

日報要約は次の**3種類**を用意する（APIは「全員用」「代表日報」「両方」を切り替え可能）。

| 種類 | 対象 | 用途 |
|------|------|------|
| **全員用のみ** | 全員用日報だけを集約・要約 | 個人の声・体調・体感の傾向を見る（必要時にヘッダー等から参照） |
| **代表日報のみ** | 代表日報（時間帯別）だけを集約・要約 | 時間帯別の店の状況の流れを見る（必要時に参照） |
| **全員用＋代表日報（統合）** | 両方を含めて**1本**に要約 | **店長・マネージャー日報に張り付ける用**。その日の全体像を1ブロックで把握 |

※ **店長日報に張り付けるのは「全員用＋代表日報」の統合1本**（基本B）。

### 7.1 要約での出所表示（元データ識別子）— 後回し・AI設定時に検討

- AI要約を構築する際、**元データの出所を要約文に含める**案がある。例：`在庫チェックについて、自動化システムを導入してほしい。zenin_kobayashi_260210` のように、発言・記述の末尾に識別子（例：種別_記入者_日付など）を付けると、本部が元データをたどりやすくなる。
- この仕様は**後回し**とし、**AI設定（プロンプト・出力形式の設計）の時に改めて検討**する。

## 8. 実装時の検討事項（メモ）

1. **既存の日報との関係**  
   **一般スタッフ日報**は「全員用」に置き換える想定。**店長・マネージャー日報は残す**（時間帯日報・全員日報を参照してまとめを書く）。
2. **時間帯マスタ**  
   別途定義・議論する。ランチ／アフタヌーン／ディナーの名称・開始・終了を設定で持つか、固定か。
3. **代表者記入欄**  
   店長・マネージャーが該当日のシフトを参照しながら、時間帯ごとの代表を選んで保存する画面を実装する。
4. **全員用の「その日出勤者」**  
   誰が出勤していたかはシフトデータと突き合わせるか、全員用を出した人＝出勤した人とするか。
5. **店長日報**  
   入力時の参照は**元データのみ**。保存時に**全員＋時間帯を統合した1本の要約**を生成して日報に張り付ける（基本B）。日別売上は look up（閲覧時に参照して表示）。本部は店長日報を見れば全て把握できる。要約内の出所表示（元データ識別子）は後回し・AI設定時に検討。
6. **ダッシュボード・ヘッダー**  
   ダッシュボードには店長・マネージャー日報のみ。全員日報・時間帯日報はヘッダーから必要時にアクセス。本部ヘッダーは全権限を網羅（現状確認・不足なら追加）。
